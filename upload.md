# Patient Bulk Upload Review

## Current Implementation

The patient upload feature is already partially implemented and is in a good direction. The current flow is:

- `DataUploadController@index` renders `resources/js/pages/data-upload/index.tsx`.
- `DataUploadController@patientTemplate` downloads an Excel template through `PatientImportTemplate`.
- `DataUploadController@importPatients` validates the uploaded file with `ImportPatientsRequest`, resolves the active branch, and delegates the work to `ProcessPatientImport`.
- `ProcessPatientImport` creates a `PatientImport` instance and calls `Excel::import()`.
- `PatientImport` uses Laravel Excel with heading rows, validation, skipped failures, and chunk reading.
- Feature tests exist in `tests/Feature/Controllers/DataUploadControllerTest.php`.

This matches the project architecture well because the controller stays thin and the import logic lives in an Action/import class.

## What Is Good

- The upload has a dedicated controller endpoint instead of being mixed into patient registration.
- The business flow is delegated to `ProcessPatientImport`, which follows the Action pattern used by this application.
- `maatwebsite/excel` is already installed, so the feature supports `.csv`, `.xlsx`, and `.xls`.
- The template is generated by code, not stored as a manual file.
- The import uses `WithChunkReading`, which is better for larger spreadsheets.
- Invalid rows are skipped and reported back to the user instead of crashing the whole import.
- The frontend uses the generated Wayfinder controller action for upload and template download.
- Feature tests cover rendering, permissions, template download, successful import, validation failures, duplicate phone numbers, missing file validation, and displaying import results.

## Corrections To Make First

### 1. Require An Active Branch Before Importing

Status: implemented.

`DataUploadController@importPatients` now fails early if there is no selected active branch. This prevents patient numbers from being generated with an empty branch context.

### 2. Fix Duplicate Detection Within The Same Uploaded File

Status: implemented in the import validator.

The import now tracks normalized phone numbers already seen in the uploaded file and rejects later rows that repeat the same phone number.

### 3. Decide Whether Email Is Tenant-Scoped Or Globally Unique

Status: implemented as tenant-scoped.

Patient email uniqueness is now scoped to `tenant_id + email` in both validation and database constraints.

### 4. Handle Excel Date Cells Safely

Status: implemented.

The import now normalizes `date_of_birth` before validation, including Excel serial date values.

### 5. Align The Validation Message With Supported File Types

Status: implemented.

The required-file message now says CSV or Excel.

### 6. Improve Row Error Names

Status: implemented.

Import errors now use the patient's name and phone number when those values are available.

### 7. Be Explicit About Partial Import Behavior

The current implementation imports valid rows and skips invalid rows. That is acceptable, but the UI and documentation should make this rule very clear.

Recommended approach:

- Keep partial import for now because it is friendlier for migration work.
- Add an error report download so staff can fix only the failed rows.
- If the product later needs all-or-nothing imports, validate all rows first and commit inside a transaction only after the user confirms.

## Suggested Next Implementation Steps

1. Add an error report CSV download for failed rows.
2. Add a preview step before committing rows.
3. Store an import audit record with uploaded filename, user, branch, imported count, skipped count, and timestamps.
4. Add queue support for large imports.
5. Add import pages for the other setup catalogs listed below.

## Other Upload Areas To Build

Patient import is only one part of the setup story. These modules should eventually get their own templates, validation rules, preview screens, and import actions.

### Facility Services

Useful for loading the hospital charge master: consultations, procedures, ward fees, theatre charges, nursing services, and other billable services.

Suggested columns:

```text

name
category
cost_price
selling_price
is_billable
```

Important checks:

- Service code must be unique per tenant.
- Category must match the existing service category enum.
- Prices must be numeric and non-negative.
- Existing services should be updated only through an explicit update mode.

## Recommended Target Flow

The safest long-term healthcare import flow is:

1. Download patient template.
2. Fill spreadsheet.
3. Upload file.
4. Server validates every row and returns a preview.
5. User reviews valid rows, errors, and duplicate warnings.
6. User confirms.
7. Server commits valid rows and stores an import audit record.
8. User can download an error report for failed rows.

The current implementation is a solid first version of steps 1, 2, 3, 6, and partial 8. The next major improvement should be the validate-and-preview step before records are created.

## Testing Checklist

Keep or add focused tests for:

- Permission required for template download and import.
- Upload without an active branch fails cleanly.
- CSV upload imports valid patients.
- XLSX upload imports valid patients.
- XLSX date cells are normalized correctly.
- Invalid enum values are skipped and reported.
- Duplicate phone already in database is skipped.
- Duplicate phone inside the same file is skipped.
- Cross-tenant duplicate phone behavior is correct.
- Cross-tenant duplicate email behavior matches the chosen product rule.
- Missing file, wrong file type, and oversized file are rejected.
- Import result appears on the data upload page.
- Failed rows can be downloaded as an error report once that feature is added.
